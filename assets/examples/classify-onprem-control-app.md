# Worked Example — On-premise control utility paired with a connected device

A common pattern: a hardware manufacturer offers a connected device, and a desktop or mobile application that runs on the user's own machine (no cloud infrastructure of the manufacturer) and that is used to configure the device. The application affects device function.

---

## Facts (hypothetical)

- A company manufactures a connected industrial sensor.
- The company also distributes a free desktop application that the user installs on the user's own machine.
- The application is used to configure measurement parameters, calibrate the sensor, and export measurement data to local files.
- All processing happens on the user's machine; no cloud infrastructure is involved.
- The company does not have a contractual relationship with users that includes a SaaS subscription. The application is shipped with the device.

## Classification

### Q1 — Connected product (Art. 2(5))

- (1a)–(1c) Likely YES on the assumed facts (sensor generates use / environment data; can communicate via network or USB; primary function is measurement).

**Conclusion:** the sensor is a connected product.

### Q2 — Related service (Art. 2(6))

- (2a) The desktop application is a digital service / software. **YES.**
- (2b) The application is connected with the sensor at the time of sale (shipped together). **YES.**
- (2c) The application configures and calibrates the sensor — it adapts and can update / change the sensor's functions. **YES.**

FAQ Q10 indicators: bidirectional data exchange (YES — application reads measurements from sensor and writes configuration); affects product function (YES). Both indicators support classification.

**Conclusion:** the application is a related service.

Note: form factor (on-premise desktop vs cloud SaaS) does not change the related-service analysis. Article 2(6) covers any "digital service, including software."

### Q3 — DPS (Art. 2(8))

- (3a) Provided to a customer? — There may not be a contractual relationship treating the user as a "customer" of the application separately from the device sale. The application is shipped with the device and runs on the user's own machine.
- (3b)–(3d) The application is **not** ubiquitous, on-demand cloud-hosted compute. It runs locally.

**Conclusion:** the application is **not** a DPS. Chapter VI does not apply.

### Q4 — Overlap

Q2 YES, Q3 NO. **Chapter II only.**

### Q5 — Article 31 carve-outs

Not relevant — not a DPS.

[As applicable.]

## Outputs the manufacturer must produce

| Output | Article | Side | Status |
|--------|---------|------|--------|
| Pre-contract notice for connected product | Art. 3(2) | Data Holder | Required at sale |
| Pre-contract notice for related service | Art. 3(3) | Data Holder | Required when bundling the application |
| Indirect-access mechanism for product data + related-service data | Art. 4(1) | Data Holder | Required (12 Sep 2025) |
| Trade-secret confidentiality agreement template | Art. 4(6) | Data Holder | Available where applicable |
| Public switching info + register | Arts. 26, 30 | — | Not required (no DPS) |
| Public Art. 28 jurisdiction + measures page | Art. 28 | — | Not required (no DPS) |

**Important nuance for related-service data on-premise.** Because the application runs on the user's own machine, the manufacturer (as data holder) may not have the related-service data unless it is uploaded back. If the manufacturer does not collect any related-service data, the Article 3(3) disclosure is correspondingly minimal — but the Article 3(2) connected-product disclosure for the sensor still applies in full.

## Open questions the memo flags

- Does the application phone home / upload telemetry? If yes, the manufacturer becomes data holder for that telemetry as related-service data, and Article 3(3)(b) volume estimates apply.
- Does the device have on-device storage from which the user can retrieve data without contacting the manufacturer? If yes, "direct access" for that data may be the chosen mode under Article 3(1).
- Is calibration data within the user's access right? Generally yes for calibration metadata generated by use; manufacturer's master calibration recipe is more likely IP / trade secret.

## Useful contrast with the SaaS-controls-device example

| Aspect | On-premise utility (this example) | SaaS that controls device (other example) |
|--------|------------------------------------|--------------------------------------------|
| Connected product | Same | Same |
| Related service | Yes | Yes |
| DPS | No | Yes |
| Number of public pages required (Arts. 26, 28, 29) | None | Three |
| Two-flow architecture (Chapter II vs VI) | Not needed | Recommended |
| Switching obligations | None | Full Chapter VI |
